# fairmind-loop

Run a Fairmind **task or user story** in **loop mode**: define an objective, machine-verifiable stop condition up front, then let an *executed* gate drive implement‚Üíverify‚Üíiterate until the checks pass ‚Äî instead of prompting every turn. Interactive mode is unchanged; this is the opt-in loop driver.

## Usage

```bash
/fairmind-loop                # loop on the active Fairmind task/story
/fairmind-loop US-142         # loop on a specific story/task ref
```

## What it does

This command is the driver. The **gate** is the `loop-check.sh` Stop hook, which runs `run_gate_checks.py` on every stop attempt ‚Äî so once the loop is armed, iteration is automatic and the loop cannot end until the stop condition holds (and the journal hook is satisfied).

## Operational quickstart (gotchas ‚Äî save the cold-start reading)

Front-loaded so you don't re-derive these from the source. Read before touching `.fairmind/` ‚Äî but **after** printing the opening banner, and **silently**: no play-by-play of reference/engine/hook reads ("let me check how the engine resolves state‚Ä¶", "let me confirm the flags of‚Ä¶"). The user-facing narrative speaks loop language (objective, check, admission, budget, arm, iterate), never engine internals; the first substantive thing the user reads after the banner is the contract summary.

- **`$CLAUDE_PLUGIN_ROOT` is empty in your (orchestrator) shell ‚Äî resolve it before running any script below.** Claude Code substitutes it in the `allowed-tools` frontmatter and sets it inside the plugin's own hooks, but it is **not** exported to the Bash calls you issue from this command body, so a literal `python3 "$CLAUDE_PLUGIN_ROOT"/scripts/‚Ä¶` runs as `/scripts/‚Ä¶` and errors. Resolve the install path once and use that absolute path in every invocation below:

  ```bash
  python3 -c "import json,os;print(json.load(open(os.path.expanduser('~/.claude/plugins/installed_plugins.json')))['plugins']['fairmind-coding@fairmind-plugins'][0]['installPath'])"
  ```

  That absolute path is exactly what CC expands `$CLAUDE_PLUGIN_ROOT` to in `allowed-tools`, so using it keeps every call inside the granted permissions. (When self-hosting inside the plugin repo itself, the repo copy `plugins/fairmind-coding/scripts/‚Ä¶` also works.)
- **State discovery needs `base_path`.** The Stop-hook gate runs `run_gate_checks.py --cwd <repo>` with **no `--state`**, so it finds `loop-state.json` only via `.fairmind/active-context.json`'s **`base_path`** field (or the `FAIRMIND_BASE` env). Bootstrap `active-context.json` **with** `base_path` (e.g. `".fairmind"`) pointing at the folder holding `loop-state.json`, or the gate silently **no-ops** (exit 0 = "no active loop") and the loop never enforces.
- **A prior run's `active-context.json` is stale ‚Äî repoint it, don't inherit it.** In a repo already used for a loop, `active-context.json` still carries the previous task's `base_path`/`task_ref`. If you inherit them, the gate resolves the *old* loop's `loop-state.json` (worst case a terminal prior loop ‚Üí the gate no-ops ‚Üí the new loop never enforces) and every pre-repoint trace op is misattributed to the old loop's ledger. The repoint is an **engine step**, not a manual chore: run `loop_open.py --repoint --task-ref <ref> [--base-path <path>]` as the run's **first mutation** in Phase 0 (before any other `.fairmind/` write). It atomically read-merge-writes `active-context.json`, setting `task_ref` (and `base_path` when given) while preserving every other field, and bootstraps the minimal shape if the file is absent. Still surface the stale-context warning to the user when `active-context.json` names a different `task_ref` ‚Äî but the repoint itself is the engine step, not a hand-edit you can forget.
- **Admission is live-RED, and the engine records the proof ‚Äî nobody pre-writes it.** `admit_check.py` re-runs the check *now* and requires it to actually fail on the current code ‚Äî author the check and admit it **while the code is still red**, before the maker fixes anything. `source.red_first_proof` is **optional at spec time**: when the checker's descriptor omits it (the normal case ‚Äî the check doesn't exist yet when it's specified), a live-RED result at admission is what admits it, and `admit_check.py` **writes** `red_first_proof` (the observed value + the current commit) into `loop-state.json` itself. Do not hand-author `red_first_proof` before admission; if it's missing, that is expected, not a defect.
- **`--dry-run` evaluates regardless of `status`.** It never persists (no `save_state` call), so there is nothing to protect by refusing it on a not-yet-armed loop ‚Äî the arm-time smoke (step 7) runs the full gate on the loop while it is still `specified`, before `--arm` is ever called. This is the read-only verb only: a plain invocation (no `--dry-run`, the shape the Stop hook uses) still refuses on any `status != "running"`, and a repo with no `loop-state.json` still no-ops either way ‚Äî the loop stays an inert signal outside itself.
- **`validate-fairmind-path.sh` scopes `.fairmind/` writes to `active-context.json`'s `base_path`** (plus the root `active-context.json` itself) ‚Äî a `..` segment or any path outside that scope is refused. Run the Phase 0 repoint (`loop_open.py --repoint`, which sets `base_path`) **before any other `.fairmind/` write** this run; once it has run, Write/Edit to `${FAIRMIND_BASE}/‚Ä¶` passes normally.
- **Two enforcement points, on two different events.** `loop-check.sh` (the gate) fires on **Stop** ‚Äî the main orchestrator's turn ends only if the gate passes. `check-journal.sh` (the journal) fires on **SubagentStop** ‚Äî a sub-agent that mutated non-`.fairmind/` code cannot finish until it writes a journal. (PCF-1: the journal hook was on `Stop` and gated on `CLAUDE_AGENT_NAME`, a variable the harness never sets, so it enforced for nobody; `Stop` never fires for a sub-agent at all. It now reads the sub-agent's `agent_type` from the `SubagentStop` payload and enforces for **any** code-mutating sub-agent, not a role allowlist. The main orchestrator never triggers `SubagentStop`, so it is still never journal-gated.)

### Phase 0 ‚Äî Contract & budget (Technical Lead, one human touchpoint)

**Opening (always the first user-visible output).** Before any preparation reading, capability probing, skill loading, or bootstrapping, print the opening in two steps, in one message:

1. Run the opener ‚Äî it prints the **banner** and, on a first run in this repo, the stacked-loop **map** (it decides that from `.fairmind/active-context.json`, so `/fairmind-loop` reads as loop 2 nested in Fairmind's outer loops):

   ```bash
   python3 "$CLAUDE_PLUGIN_ROOT"/scripts/loop_open.py
   ```

   **Show that output as-is; do not re-write, summarise, or repeat it** ‚Äî it is a script's stdout: byte-identical, one tool call instead of re-emitting ~30 lines.

2. Render the **transactional dashboard** and **include its output in your message so it renders as a table**:

   ```bash
   python3 "$CLAUDE_PLUGIN_ROOT"/scripts/loop_dashboard.py --md
   ```

   It is a Markdown table ‚Äî an active-loop header + one row per loop (status, checks, iter, tier, tool-calls by kind, duration, best-effort tokens) + totals. Relay it **verbatim**: the script generates the exact table, so it stays deterministic; do not hand-edit it. A fresh repo shows an empty dashboard ‚Äî that is correct, not an error. (This is the **run** dashboard; the **trace** ledger of tool ops under `.fairmind/trace/‚Ä¶` is a different artifact, summarised only in the Exit report.)

Then do the machinery reading **silently** ‚Äî no play-by-play of engine/hook reads; the next thing the user reads is the contract summary.

1. Engage **the Technical Lead** to resolve context and bootstrap `.fairmind/` (Phase 0 + Phase 0b). The loop runs on **any repo, zero-config**, and the run's **first mutation** is the repoint ‚Äî it both bootstraps an absent `active-context.json` and rescues a stale one from a prior loop, atomically, so no op that follows (the gate resolving `loop-state.json`, `trace-op.sh` naming the trace file) can land on the previous loop's `base_path`/`task_ref`:

   ```bash
   python3 "$CLAUDE_PLUGIN_ROOT"/scripts/loop_open.py --repoint --task-ref <task-or-story-ref> --base-path .fairmind
   ```

   This writes the minimal shape the runtime needs ‚Äî **`base_path`** (how the Stop-hook gate finds `loop-state.json`, per the quickstart gotcha above) and **`task_ref`** (how `trace-op.sh` names the trace file it attributes each mutation through; a missing `task_ref` silently costs every agent its attribution) ‚Äî while preserving every other field already present:

   ```json
   {"mode":"loop","fairmind":"none","project":"<repo-name>","base_path":".fairmind","task_ref":"<task-or-story-ref>"}
   ```

   Run it **before any other `.fairmind/` write** this run. When `active-context.json` already exists with a different `task_ref` (a prior loop), also **warn the user** that the stale context is being repointed ‚Äî the repoint is the engine step that makes it safe, not a hand-edit you can forget. The Technical Lead then asks the user **once** whether a Fairmind workspace exists, recording the answer in the `fairmind` field (`none` | `configured`) ‚Äî never re-asking while it stands. If yes, it points to Studio ‚Üí avatar ‚Üí **Developer** page for the API key + MCP snippet (until `/fairmind-connect` ships in F4).

   - **Optional isolated worktree.** The Technical Lead may offer to run the maker's edits in a dedicated git worktree instead of the user's own checkout ‚Äî **opt-in**, and least-urgent while this plugin is self-hosting its own tasks. If offered and accepted, create it before the maker's first turn:
     ```bash
     python3 "$CLAUDE_PLUGIN_ROOT"/scripts/loop_worktree.py --create --task-ref <ref> --state "${FAIRMIND_BASE}/loop-state.json"
     ```
     Maker edits then land on branch `loop/<ref>` inside that worktree ‚Äî the user's checkout stays untouched ‚Äî and the exit report points at the `loop/<ref>` branch diff instead of the working tree. It is idempotent (safe to re-run; reuses the same worktree). Tear it down once the loop closes:
     ```bash
     python3 "$CLAUDE_PLUGIN_ROOT"/scripts/loop_worktree.py --cleanup --task-ref <ref>
     ```
     Once `worktree.{path,branch}` is recorded, the gate **follows** it: every check verdict and the `contract.scope` boundary are evaluated against the worktree, not the user's own checkout (see `fairmind-gate/references/loop-state.md`'s `worktree` section). If the recorded path stops resolving to a real, registered worktree of this repo (removed on disk, or `--cleanup` ran without clearing `state["worktree"]`), the gate fails the whole evaluation closed to `blocked_worktree` rather than silently falling back to the main tree ‚Äî see the Exit section below.
2. The Technical Lead **classifies** each acceptance criterion / objective into a check type and **specifies** descriptors into `${FAIRMIND_BASE}/loop-state.json` (using the `fairmind-gate` skill). The Technical Lead never writes check code. It also **persists the classified criteria into `contract.criteria[]`** ‚Äî one entry per acceptance criterion, `{ id, text, disposition, hard }`, where `disposition` names the check that covers it (`checked:<id>` / `evidence:<id>` / `quarantined:<id>` / `unverifiable`) and `hard` defaults to `true` (see `fairmind-gate/references/loop-state.md`). **This is mandatory for arming, never inferred**: `--arm` refuses a loop whose `contract.criteria` is absent, null, or empty, and refuses any HARD criterion not covered by a live, admitted check ‚Äî so a Phase 0 that skips this step produces a loop the engine will not arm. Optionally, the Technical Lead may also declare `contract.scope.allowed_paths` (a glob list) ‚Äî the run's allowed mutation footprint. When declared, every gate evaluation cross-checks the run's git-derived mutation set against it *before* any check verdict is considered; a mutated path outside scope is a **terminal hard stop** (`status: "blocked_scope"`), never an ordinary red and never counted against budget. No declaration ‚Üí identical to today (no-op). **A declared scope must cover *every* path a mutation lands on while the loop is `running` ‚Äî not only the makers' code.** Only `.fairmind/` is auto-exempt (the loop's own workspace, dropped structurally by `_is_loop_workspace_path`); `--arm` additionally exempts files *already dirty at arm time* (`contract.mutation_set.baseline.pre_dirty`, anchored to their arm-time content). Anything written **fresh** while `running` ‚Äî a run/findings log under `docs/`, a scratch artifact, a generated file ‚Äî is a git-visible mutation and trips `blocked_scope` unless its path is in `allowed_paths`. So when you (orchestrator) or a maker will write to a path outside the code surface during the run ‚Äî e.g. appending to this repo's internal log while internal ‚Äî add that path to `allowed_paths` at spec time (finding F2). **A pre-compiled contract may be consumed instead of hand-authoring this step.** When the task originates from an external ticket run through `/loop-import`'s `--emit` (`task-compilation` skill + `scripts/loop_import.py`), its output is already a `loop-state.json` with `checks[]` and `contract.criteria[]` populated ‚Äî reuse it (and its `.fairmind/contracts/<ref>.json` copy) as the starting point for this step instead of classifying from scratch, still confirming admission and coverage before arming.
3. The Technical Lead detects capabilities ‚Üí **hermeticity tier** (`srt` present ‚Üí A, else B), and **captures baselines** for reduce/improve goals.
4. Check authoring is **delegated to the checker-side, RED-first** (the QA Engineer for functional/evidence, the Code Reviewer for metric/performance, `sonar_gate.py` for static) ‚Äî **maker ‚âÝ checker**. Each authoring dispatch is **stateless and self-contained**, following `fairmind-gate/references/authoring-brief.md`: full descriptor inline, the four admission gates as acceptance criteria, exact output format, no scope expansion, and the INPUT GAP rule (missing input ‚Üí say so, don't guess).
5. Run **admission**: `python3 "$CLAUDE_PLUGIN_ROOT"/scripts/admit_check.py --state "${FAIRMIND_BASE}/loop-state.json"`. Quarantined checks are surfaced, excluded from stop. **A non-zero exit blocks arming**: `admit_check.py` exits non-zero when every check it considered ended up quarantined, or when it considered zero checks at all (an admission run that admits nothing is never a success) ‚Äî do not proceed to Phase 0b step 8 on a non-zero exit; fix or re-author the offending check(s) and re-run admission first. A partial quarantine (at least one admitted check) still exits 0 and arming may proceed, with the quarantined check(s) surfaced to the human.
6. The Technical Lead **proposes a budget** sized to the task and **asks the user to confirm or adjust it**. Default shape:
   - `max_iterations ‚âà 2 √ó (number of admitted checks)` ‚Äî headroom for a red-then-fix cycle per check.
   - `max_consecutive_failures = 3` ‚Äî a check that fails three evaluations running is a strategy problem, not a budget problem.
   - `timeout_min` sizes the whole **arm‚Üíclose** wall clock, not a single gate evaluation. Since T19, `--arm` stamps `budget.spent.started_at` at arm time, so the timeout clock runs across every maker/checker turn between arming and the loop's terminal transition ‚Äî not just the gate's own (sub-second) execution. Size it from a realistic estimate of wall-clock time per iteration (dispatch + implement + verify), e.g. `timeout_min ‚âà typical wall-clock minutes per iteration √ó (max_iterations + K)`, rounded up with generous headroom. The old single-evaluation-wall-time formula would trip `blocked_timeout` at the very first evaluation of every loop once the clock starts at arm ‚Äî size for the **arm‚Üíclose** span, not the gate's own runtime.

   Do not arm the loop until the user confirms.
7. **Arm-time smoke run:** the loop is still `specified` at this point ‚Äî Phase 0b's bootstrap status, never yet armed. Run the full gate once via `--dry-run` and require **zero ERROR/INCONCLUSIVE** results ‚Äî a missing runner, an unreadable report, a non-deterministic signal must surface now, at arm time, not on iteration 1 of a live loop. `--dry-run` evaluates the gate regardless of `status` and never persists, so it runs cleanly on the `specified` loop with no hand-edit required:
   ```bash
   python3 "$CLAUDE_PLUGIN_ROOT"/scripts/run_gate_checks.py --dry-run
   ```
   If any check is ERROR or INCONCLUSIVE, fix the environment/descriptor (or quarantine the check) and re-run the smoke before arming.
8. **Start gate:** optionally inspect coverage read-only first, then arm with the engine's own verb ‚Äî never hand-edit `loop-state.json.status`:
   ```bash
   python3 "$CLAUDE_PLUGIN_ROOT"/scripts/run_gate_checks.py --validate-contract   # read-only: is this contract armable?
   python3 "$CLAUDE_PLUGIN_ROOT"/scripts/run_gate_checks.py --arm
   ```
   Arm only when the budget is confirmed **and** the arm-time smoke run is clean. `--arm` runs the contract validation **internally** (its third refusal gate, after already-`running` and no-admitted-check) and **refuses an invalid contract**: every HARD `contract.criteria[]` entry must be covered by a live, admitted check, or `--arm` exits non-zero, names each uncovered criterion on stderr, and leaves `loop-state.json` byte-untouched (status not flipped, `started_at` not stamped, no arm audit entry). `--validate-contract` is the read-only twin that reports the same coverage without side effects ‚Äî the verb a human runs to ask "is this armable?".

   **Decline-to-arm branch.** If a task's hard criteria cannot be covered by a live, admitted check ‚Äî the check does not exist, the check is quarantined, or the criterion is genuinely `unverifiable` ‚Äî do **not** arm a weak gate. The engine refuses, and the recommendation (from the engine's own stderr, and here) is to run the task in **interactive** mode instead, or to downgrade a genuinely advisory criterion to `hard: false` (which the coverage report then names as an accepted hole). A loop whose stop condition cannot be gated is not a loop; arming it anyway is exactly the failure this validation exists to prevent. Arming never proceeds on the orchestrator's own judgement of coverage ‚Äî the engine decides, on the persisted contract.

   On a successful arm, `--arm` flips `status` to `running`, stamps `budget.spent.started_at` from its own clock (so the measured window starts when the work starts, not at the first gate evaluation), and zeroes `confirmations` ‚Äî the same verb re-arms a `blocked_*`/`passed_pending_human` loop after a human gate rejection, preserving its original `started_at` and clearing `owner_session` so a fresh session can claim it. A non-zero exit means arming was refused (already `running`, no admitted check, or an invalid contract) ‚Äî read the stderr reason before retrying. Once armed, render the dashboard (`python3 "$CLAUDE_PLUGIN_ROOT"/scripts/loop_dashboard.py --md`, included in your message as a table) so the user sees the loop go live with its active-loop header.

### Phase 1..N ‚Äî Iterate (automatic via the gate)

- **Maker turn:** The Software Engineer reads `${FAIRMIND_BASE}/steering.md` first, if present (the human steering channel ‚Äî see Notes below), then implements (composing tech skills + `fairmind-tdd`), reads any prior gate feedback, and journals per iteration under the **apply-or-rebut** rule: every non-green item is either **APPLIED** (the code changed ‚Äî say what) or **REBUTTED** (why it is wrong ‚Äî flagged to the checker, never by editing the descriptor).
  - **Drive the maker to completion within the turn.** The gate fires on *every* orchestrator turn-end. If the maker is dispatched as a background/async `Agent` that keeps writing after the turn ends, the gate evaluates a **half-written tree** ‚Äî charging budget and firing spurious STRATEGY TURNs on merely-incomplete work, or advancing the confirmation streak on a *false green* whose RED-making test has not landed yet. So either dispatch the maker **synchronously**, or **hold the orchestrator turn open** until the maker's artifacts are complete (block on it ‚Äî e.g. `TaskOutput(block:true)` ‚Äî before ending the turn) so the gate only ever judges a finished tree. The engine has a **settle-window safety net** (H8): an evaluation whose most recent work-product write landed within the last `FAIRMIND_GATE_SETTLE_S` seconds (default 45) is treated as *in-flight* and does not move budget, `consecutive_failures`, or the confirmation streak ‚Äî but that is a backstop for a forgotten hold, not a substitute for driving the maker to completion.
- **Gate turn:** on every stop, `loop-check.sh` runs the checks. Not green + budget left ‚Üí the hook blocks the stop and feeds structured feedback (routed to the owner) as the next turn's input. All green for **K ‚â• 3 consecutive** evaluations ‚Üí `status: passed_pending_human`.
  - **Reaching K on an already-green, settled tree does not require K empty turns (finding F3).** The Stop hook ticks the streak once per turn-end, but a real evaluation *is* just `run_gate_checks.py` (no `--dry-run`) ‚Äî the same invocation the hook makes. So when the maker has landed a green tree and it is **stable** (nothing left to write), you may run that command directly, back-to-back, to accumulate the confirmation streak to K in a single turn, instead of ending K near-empty turns to let the hook tick. Each such run is a genuine, state-mutating evaluation (it advances `confirmations`), and the H8 settle window still applies ‚Äî an evaluation within `FAIRMIND_GATE_SETTLE_S` (default 45s) of the last work-product write is treated as *in-flight* and will **not** advance the streak ‚Äî so drive it only once the tree has genuinely settled, never to paper over a maker that is still writing. And mind the shell timeout when a check runs the full test suite: at tens of seconds per evaluation, K back-to-back runs in one Bash call can outlive the Bash tool's default timeout (currently 120s) and the kill lands mid-evaluation ‚Äî completed evaluations keep their persisted confirmations (recoverable, just wasteful) ‚Äî so either raise the Bash timeout for the K-drive or run the evaluations one per command.
- **Commitment boundaries (warn-and-route, never auto-quarantine):** the gate watches iteration history for two stall signals. A check at **2 consecutive failures** (one below the cap) raises a **STRATEGY TURN**, routed once to the *checker* ‚Äî is the check wrong, or the approach? Two checks **strictly anti-correlated** over the last 3 evaluations raise a **CONTRACT CONFLICT**, routed to the *Technical Lead* to reconcile the two criteria. Both only warn and re-route; the human decides.
- You can inspect without mutating state at any time:
  ```bash
  python3 "$CLAUDE_PLUGIN_ROOT"/scripts/run_gate_checks.py --dry-run
  ```

### Exit ‚Äî final human gate

- **Green confirmed ‚Üí `passed_pending_human`.** Present the report: what passed, each check's hermeticity tier, iteration count, advisory/quarantined checks. **No auto-merge/deploy** ‚Äî a human approves, then the Fairmind task status is updated.
- **Adversarial pass before the human, for a gate/engine change.** If this loop changed the gate's own safety logic, a green suite is necessary but not sufficient ‚Äî run an independent "try to break each fix" pass (revert-on-scratch to prove each check has teeth; drive the real engine on the failing combinations) **before** presenting to the human. See `fairmind-gate` ‚Üí "Authoring lessons the engine can't enforce". It has repeatedly caught green-but-wrong fixes a green gate passed.
- **Budget exhausted ‚Üí `blocked_*`.** Present the report and flag for the human. The loop never runs indefinitely. The report ends with a **cost-ask** ‚Äî the remaining red checks, the trend, and the exact `--extend-budget` command.
- **Scope violation ‚Üí `blocked_scope`.** Only when `contract.scope.allowed_paths` was declared (step 2): a mutated path outside it hard-stops the loop, naming the path and its attributing agent ‚Äî this is not a budget or check failure, it never consumed either. Present it for the human to reconcile (revert/relocate the out-of-scope mutation, or amend the declared scope) before re-arming; do not treat it as a check to fix and iterate past.
- **Worktree degraded ‚Üí `blocked_worktree`.** Only when `worktree.{path,branch}` was recorded (step 1's optional isolated worktree): if the path stops resolving to a real, registered worktree of this repo, the gate refuses to evaluate anything against the main tree instead and hard-stops here ‚Äî again not a budget or check failure. Present it for the human to resolve ‚Äî re-create the worktree (`loop_worktree.py --create --task-ref <ref>`) or clear `state["worktree"]` ‚Äî before re-arming.
- **Dashboard:** render `python3 "$CLAUDE_PLUGIN_ROOT"/scripts/loop_dashboard.py --md` and include the table in the closing report ‚Äî this loop now shows as its finalized üü¢/üî¥ row with its tool-call + token stats and the run totals (the engine appended the ledger row on the terminal transition).
- **Presenting the trace ledger:** when the report covers `.fairmind/trace/<taskRef>.jsonl`, summarize it as a digest ‚Äî total ops, breakdown by `kind` (`mutate`/`exec`/`dispatch`/`read`), and the mutated files with the agent that touched each. Never paste raw JSONL into the report; give the file path, and show raw lines only if the user explicitly asks for them.
- **Agentic Insights terminal flush, on close.** Once the loop reaches a terminal status, flush what this close produced to project-context: run `python3 "$CLAUDE_PLUGIN_ROOT"/scripts/insights_flush_payload.py --emit all` and parse its `{"loop": ..., "decisions": ...}` output. For each category whose value is non-null, call the matching MCP tool ‚Äî `mcp__Fairmind__Insights_record_loop_stats` for `loop`, `mcp__Fairmind__Insights_record_agent_decisions` for `decisions` ‚Äî then run `python3 "$CLAUDE_PLUGIN_ROOT"/scripts/insights_flush_payload.py --commit <category>` **only after that category's MCP call succeeds** (`--commit all` only once both have). A failed or partial send must stay **uncommitted**: do not commit a category whose call errored or did not complete ‚Äî it stays unflushed on disk and `/fairmind-sync-insights` retries it later, so `--commit` never marks unsent data as sent. If the Fairmind MCP is not connected at all, skip the calls the same way: the payloads stay on disk unflushed, and `/fairmind-sync-insights` picks them up later ‚Äî a mode, not an error.
- **Studio task-lifecycle write-back, only after the human approves.** Once the human approves a `passed_pending_human` loop ‚Äî never automatically on a green gate, mirroring the no-auto-merge rule above ‚Äî reflect the result onto the Studio task itself, attempting the two calls independently so a failure in one never masks the other: `<task_id>` is the Studio task id that the loop's `task_ref` resolves to (the ref set by Phase 0's repoint into `active-context.json`/`loop-state.json`), and `<project_id>` is the project from that same `active-context.json`. `mcp__Fairmind__Studio_bulk_update_status(ids=[<task_id>], status="passed", entity_type="task")` advances the task's status, and `mcp__Fairmind__Studio_process_journal(task_id=<task_id>, journal_content=<journal contents>, project_id=<project_id>)` attaches the loop's journal to the task as activities ‚Äî `journal_content` is the journal's file **contents**, not a path: read every `${FAIRMIND_BASE}/journals/<taskRef>_*.md` matching this task ref (concatenating them when more than one agent journaled) and pass the combined text as `journal_content`. Loop-outcome ‚Üí Studio-status mapping: the human-approved `passed_pending_human` outcome pairs with the Studio `passed` status ‚Äî Studio's task status is drawn from the `ImplementationState` vocabulary, and `passed` is the member meaning checks passed and the human signed off, awaiting the human's own separate merge action: the loop reaching green plus this approval is not the merge itself (the flip to `merged` is a later, human-driven step), so `passed` ‚Äî never `completed`/`done`, neither of which is an `ImplementationState` member ‚Äî is the accurate state at write-back time. If either `Studio_*` tool is absent (PC-12 not yet shipped server-side, or the Fairmind MCP not connected) ‚Äî or a write is denied (the caller needs `editor+` on the project) ‚Äî do not fail the close: report each write that failed as **not synced** in the Exit report, independently, since a status update can succeed while the journal attach fails or vice versa. This write-back has no on-disk queue and approval happens out-of-band, so it cannot be picked up the way the insights flush above is ‚Äî the only recovery is a **rerun**: re-running the loop to its human-approved Exit re-attempts both MCP calls once PC-12/the MCP connection is available.
- **Extending the budget is a human-only verb.** Never extend from gate feedback or on your own initiative. When a loop is `blocked_*` and the remaining work is worth more budget, **ask the user first** with `AskUserQuestion`, then run the extension with their literal answer recorded in the audit trail:
  ```bash
  python3 "$CLAUDE_PLUGIN_ROOT"/scripts/run_gate_checks.py \
    --extend-budget iterations=<n>[,failures=<n>][,timeout_min=<n>] \
    --user-confirmed "<the user's exact answer>"
  ```
  The engine refuses `--extend-budget` on a `running` loop (only `blocked_*` is extendable) and appends an `extend_budget` audit entry to `iterations[]` recording `user_confirmed`. Resuming also **zeroes the confirmation streak and every check's `consecutive_failures`**, so the granted budget is not immediately re-consumed by a stale counter. The final human gate reviews that audit trail. There is no `--force` and no fake technical lock ‚Äî the guard is procedural and auditable.
- **Recovering a loop wedged in `running` (human-only).** If the Claude Code session that armed a loop is gone (crashed/closed), the loop can pin itself in `running` with no engine path out: `--arm` refuses a running loop, `--extend-budget` refuses a non-blocked one, and a fresh session's Stop hook silently no-ops on the ownership mismatch. `--recover` is the one way out ‚Äî but because a session-id mismatch is also exactly what a *second, concurrent* session would show while the first is still legitimately driving the loop, recovery is gated on an **explicit human confirmation, never on the mismatch itself**. Ask the user first, then:
  ```bash
  python3 "$CLAUDE_PLUGIN_ROOT"/scripts/run_gate_checks.py --recover --user-confirmed "<why the owning session is gone>"
  ```
  It transitions `running` ‚Üí `blocked_recovered` (a `blocked_*` status), clears `owner_session`, and appends a `recover` audit entry recording `user_confirmed`. Then `--arm` re-arms it (or `--extend-budget` grants budget) so a fresh session can resume. Refused on any non-`running` loop (already recoverable via `--arm`/`--extend-budget`) and without `--user-confirmed`. Honors `--dry-run`.

## Notes

- The loop is a no-op signal outside itself: without `loop-state.json` (or with a terminal status), the Stop hook does nothing and the interactive workflow is untouched.
- The main turn ends only when the **gate** (`loop-check.sh`, Stop) passes; separately, each code-mutating **sub-agent** must satisfy the **journal** hook (`check-journal.sh`, SubagentStop) before it can finish.
- **Human steering channel.** A human can drop a redirect / "don't repeat X" / a piece of context into `${FAIRMIND_BASE}/steering.md` by hand at any point in the run; the maker reads it first thing every iteration (see the "Maker turn" bullet above). It complements the gate-initiated signals ‚Äî STRATEGY TURN (routed to the checker) and CONTRACT CONFLICT (routed to the Technical Lead) ‚Äî as the human-initiated counterpart to those engine-initiated ones. Because it lives under `${FAIRMIND_BASE}` (hence under `.fairmind/`), it gets the same structural exemption as `loop-state.json`/journals/trace (`_is_loop_workspace_path`): editing it is never a code mutation, never trips `blocked_scope`, and never satisfies or breaks a check. It is user-authored only ‚Äî no script ever emits it. See `fairmind-gate/references/steering.md` for the full contract.
